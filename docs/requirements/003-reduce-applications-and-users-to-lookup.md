# 002 - Reduce `applications` and `users` to lookup-only access

## User Story

As a contributor, I want all access to application and user configuration to go through a single lookup plugin each, so that the project no longer has to generate, merge, and maintain the large `group_vars/all/05_applications.yml` and `group_vars/all/04_users.yml` dictionaries, and the complexity of the configuration surface is reduced.

## Acceptance Criteria

- [ ] The files `group_vars/all/05_applications.yml` and `group_vars/all/04_users.yml` are deleted from the repository and are not recreated by any tooling.
- [ ] The `applications` and `users` facts are instead set in the constructor stage ([tasks/stages/01_constructor.yml](../../tasks/stages/01_constructor.yml)) via `ansible.builtin.set_fact` calls that invoke the new lookup plugins (for example `applications: "{{ lookup('applications_all') }}"` and `users: "{{ lookup('users_all') }}"`), replacing the current `default_users | combine(...)` and `defaults_applications | merge_with_defaults(...)` tasks that depend on the deleted files.
- [ ] Any tooling, make target, script, or role task whose purpose is to generate, render, merge, or rewrite `group_vars/all/05_applications.yml` or `group_vars/all/04_users.yml` is removed, and running such a generator is no longer possible from the project.
- [ ] The deprecated `make setup` flow no longer generates the two files: the corresponding steps in [scripts/setup.sh](../../scripts/setup.sh) that write to `APPLICATIONS_OUT` (`group_vars/all/05_applications.yml`) and `USERS_OUT` (`group_vars/all/04_users.yml`) are removed, and the backing generators under [cli/setup/applications/](../../cli/setup/applications/) and [cli/setup/users/](../../cli/setup/users/) are fully deleted (including any validation, merge, or helper utilities they contained, which are rewritten on top of the new lookup plugins wherever still needed); if `make setup` has no remaining responsibilities after this removal, the target itself and its entry in the [Makefile](../../Makefile) are deleted, otherwise only the deprecated generation steps are removed while keeping the still-needed behavior.
- [ ] All documentation that references the removed generators (for example [docs/agents/files/group_vars/applications.md](../agents/files/group_vars/applications.md) and any contributor or agent docs that instruct running the generators) is updated to describe the new lookup-only workflow instead, so that no doc still tells the reader to regenerate the two files.
- [ ] The lookup plugin for applications returns a dict keyed by `application_id` that is built by discovering and aggregating every `roles/*/config/main.yml` in the repository; the `application_id` is ALWAYS the role directory name (for example `roles/web-app-mailu/config/main.yml` is exposed as `applications['web-app-mailu']`), and there is no alternative key source. The resulting dict MUST be overridable by entries under `applications.<application_id>` defined anywhere in inventory, group vars, host vars, or role vars, using the same precedence as before this change.
- [ ] The lookup plugin for users returns a dict keyed by `username` that is built by discovering and aggregating every `roles/*/users/main.yml` in the repository, and MUST be overridable by entries under `users.<username>` defined anywhere in inventory, group vars, host vars, or role vars, using the same precedence as before this change.
- [ ] The applications lookup returns the full set of applications across all roles, independent of `SERVICES_DISABLED` or `allowed_applications`; filtering by enabled or allowed apps remains the caller's responsibility and continues to use the existing mechanisms (for example `lookup('applications_current_play')`, which is not modified by this requirement).
- [ ] When no override is present for a given `application_id` or `username`, the lookup plugin returns the default value from the discovered role file; when no default exists, it returns a documented empty structure rather than raising, except where strict access is explicitly requested by the caller.
- [ ] Both lookup plugins cache the aggregated defaults at module level so that the filesystem scan and YAML parsing of `roles/*/config/main.yml` and `roles/*/users/main.yml` happens at most once per Ansible worker process, regardless of how many times the lookup is called. Only the defaults are cached; the merge with `variables['applications']` / `variables['users']` MUST run on every call so that overrides remain effective. Each plugin MUST expose a documented reset helper (for example `_reset_cache_for_tests()`) so that unit tests can guarantee a clean cache between test cases.
- [ ] Every reference to the top-level `applications` variable in roles, playbooks, templates, filter plugins, lookup plugins, and tests is replaced by a call to the new applications lookup (directly or via an existing wrapper such as [plugins/lookup/config.py](../../plugins/lookup/config.py)). This includes ALL access forms: dotted access (`applications.<id>`), bracket access (`applications['<id>']`), full-dict access (`{{ applications }}`), and pipeline/filter usage (for example `applications | dict2items`, `applications | combine(...)`). The only remaining reference to the bare `applications` variable is the `set_fact` in the constructor that creates it.
- [ ] Every reference to the top-level `users` variable in roles, playbooks, templates, filter plugins, lookup plugins, and tests is replaced by a call to the new users lookup. This includes ALL access forms: dotted access (`users.<name>`), bracket access (`users['<name>']`), full-dict access (`{{ users }}`), and pipeline/filter usage. The only remaining reference to the bare `users` variable is the `set_fact` in the constructor that creates it.
- [ ] The contributor documentation under [docs/contributing/](../contributing/) is updated to describe the new lookup-only access pattern, how to add or override an application entry, and how to add or override a user entry, so that no contributor is tempted to re-introduce inline data in the two reduced files.
- [ ] Unit tests cover the applications lookup and the users lookup for the default case, the override case, and the missing-entry case (strict and non-strict), and all existing tests still pass.
- [ ] Before starting implementation, [AGENTS.md](../../AGENTS.md) MUST be read and its instructions MUST be followed for the entire duration of the task.
- [ ] The implementation is iterated locally on [.github/workflows/test-environment.yml](../../.github/workflows/test-environment.yml) using the Workflow Loop defined in [iteration.md](../agents/action/iteration.md) (`ACT_WORKFLOW=.github/workflows/test-environment.yml ACT_JOB=test-environment ACT_MATRIX='dev_runtime_image:debian:bookworm' make act-workflow`) until the workflow passes end-to-end at least once without errors.
